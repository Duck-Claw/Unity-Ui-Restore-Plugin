Unity 界面还原插件可行性评估

基于切图 PNG 与 `export_manifest.json`，可以进一步做 Unity Editor 插件，自动生成 Canvas/Prefab，并支持资源改名、删减与已有组件 Prefab 替换。

评估结论

可行，建议分阶段实现。MVP 能还原布局；生产级版本必须让 JSON 记录“资产身份”，并提供映射复核与 Prefab Registry。

布局还原 高可行 manifest 已有位置、尺寸、层级路径，可生成 Unity RectTransform。
资源追踪 需增强 专业同学会重命名和删图，不能只依赖文件名。
组件复用 可支持 通过规则或登记表，将切图节点替换为已有 UI Prefab。
团队交付 值得做 关键收益是减少手工摆 UI，并输出可控节点层级。
一、输入条件与核心难点

已有输入足够做 MVP

当前 manifest 包含 `layerPath / x / y / width / height / file`，足以生成基础 Canvas 和 Image 节点。

文件名不是稳定身份

资源会被专业同学改名、删除、合并；如果 Unity 只按文件名查找，导入链路会频繁断开。

Prefab 替换是关键需求

按钮、标签、通用弹窗等不一定用 PNG，还可能直接实例化已有组件 Prefab，并继承其脚本和动效结构。

二、关键方案：JSON 标记资产身份
{ "id": "psd_9f2a_layer_1038", "assetType": "Btn", "layerPath": "Panel/Btn/Confirm", "rect": { "x": 310, "y": 612, "w": 188, "h": 72 }, "fileName": "Btn_Confirm.png", "sha256": "exact-content-hash", "visualHash": "perceptual-hash", "prefabHint": "ButtonConfirm" }

改名后仍能识别的办法

  • `id` 跟 PSD 图层绑定，文件改名也不变。
  • `sha256` 用于图片没变、只改名的精确匹配。
  • `visualHash` 用于图片被压缩或重新导出后的近似匹配。
  • `layerPath + rect + assetType` 用于重复图片的节点归属判断。
  • `prefabHint` 给 Unity 侧替换组件提供线索。

四、推荐实现路线
Phase 01 MVP 还原 读取现有 manifest，按坐标和层级生成 Canvas Prefab。
Phase 02 身份追踪 在导出阶段补 `assetId / sha256 / visualHash / zIndex`。
Phase 03 映射面板 Unity 内展示缺失、重复、疑似匹配，允许人工确认。
Phase 04 组件替换 接入 Prefab Registry，生成适合程序和动效维护的层级。

最终判断

这个插件值得做。最小版本可以快速验证还原效果；生产版本的关键不是“把图片摆回去”，而是建立资源身份、组件替换和人工复核三套机制，确保专业整理后的资源仍能稳定生成 Unity UI。

一句话原则 不要让 JSON 只记“图片叫什么”，要让它记“图片是谁、在哪里、能否被 Prefab 替换”。
