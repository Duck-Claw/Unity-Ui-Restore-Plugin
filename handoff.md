# Unity UI Restore Plugin Handoff

Last updated: 2026-05-28

## Product Summary

This is a new standalone product idea derived from PSD Auto Slicer output.

Working name: `Unity UI Restore Plugin`

Goal:

- Build a Unity Editor plugin that reads PNG assets and `export_manifest.json` generated by PSD Auto Slicer.
- Reconstruct the corresponding UI layout inside Unity as a Canvas, Scene, or Prefab.
- Support post-slicing production workflows where artists rename, delete, merge, or replace exported PNG resources.
- Support replacing certain sliced PNG nodes with existing Unity UI component Prefabs.
- Generate a clean, controllable hierarchy that programmers and motion/animation designers can use.

Feasibility verdict:

- Feasible.
- The existing manifest is enough for a layout-restoration MVP.
- A production-grade version requires stronger asset identity tracking, mapping review, and prefab replacement rules.


Current Unity entry point:

```text
Tools > PSD Auto Slicer > Restore UI
```

Current MVP features:

- Select an export folder containing `export_manifest.json`.
- Drag a Unity Project folder directly into the `Export Folder` field.
- Parse PSD Auto Slicer manifests from version `0.2.48` and newer.
- Prefer PSD Auto Slicer `0.2.49` manifests because they include identity fields and `sha256`.
- Generate a UGUI Canvas Prefab.
- Save generated Prefabs to `Assets/PSDUIRestore/Generated`.
- Convert PSD top-left coordinates to Unity centered `RectTransform` coordinates.
- Build hierarchy from `layerPath`.
- Create Unity `Image` nodes from PNG assets.
- Configure PNGs as Sprites.
- Apply Sprite borders and `Image.Type.Sliced` for nine-slice items.
- Report generated count, missing count, hash matches, and same-folder-size fallback matches.
- Support renamed PNGs if they stay in the original export folder.

Current resource matching order:

1. Original file name from manifest.
2. `sha256` exact content hash.
3. Same export subfolder plus unique PNG size.

The third fallback exists because old manifests or failed hash output may not have usable `sha256`.

Current control nodes generated at the top of the Canvas root:

```text
Bg
Top
TopLeft
TopRight
Left
Center
Right
Bottom
BottomLeft
BottomRight
```

Control node transform rules:

- `anchoredPosition = 0,0`
- `sizeDelta = 0,0`
- `Bg`: stretch / stretch
- `Top`: top / stretch
- `TopLeft`: top / left
- `TopRight`: top / right
- `Left`: middle / left
- `Center`: stretch / stretch
- `Right`: middle / right
- `Bottom`: bottom / stretch
- `BottomLeft`: bottom / left
- `BottomRight`: bottom / right

If a PSD `layerPath` starts with one of these exact top-level names, the generated content reuses that control node instead of creating a duplicate first-level group.

## Source Product Context

Upstream product:

- PSD Auto Slicer
- Compatible baseline during initial analysis: `0.2.48`
- Current local development/export version: `0.2.49`
- Upstream output includes:
  - Exported PNG files.
  - `export_manifest.json`.

Manifest fields that already help Unity reconstruction:

- `layerPath`
  Used to recover PSD-like hierarchy.
- `x`, `y`
  Used to position UI nodes.
- `width`, `height`
  Used to size Unity `RectTransform`.
- `file`, `name`, `relativeFolder`
  Used to locate exported image files.
- `layerType`, `sourceType`
  Used as basic type hints.
- `canvasSize`
  Includes original source canvas dimensions when exported with canvas-size metadata.
- `nineSlice`, `uniformSize`, `evenSizeApplied`
  Useful for import handling and sprite setup decisions.

New fields added in local PSD Auto Slicer `0.2.49`:

- root `manifestSchemaVersion: 2`
- root `sourceCanvasWidth`
- root `sourceCanvasHeight`
- item `assetId`
- item `layerId`
- item `assetType`
- item `zIndex`
- item `sha256`

The Unity plugin treats these as optional so older manifests still load. Robust rename detection is strongest when the manifest includes `sha256`.

## User-Provided Usage Context

These are expected real production workflows:

- After slicing, professional artists may rename PNG files to match project naming conventions.
- Artists may delete unnecessary PNGs, especially duplicated layer assets.
- If an exported PNG corresponds to an existing Unity UI component Prefab, artists may either keep that PNG in the folder or remove it.
- During Unity reconstruction, the tool should be able to call or instantiate existing component Prefabs instead of always creating raw Image nodes.
- The generated Unity hierarchy must be well organized so programmers and motion designers can control nodes reliably.

## Core Product Requirements

### Layout Reconstruction

The plugin should:

- Read `export_manifest.json`.
- Locate usable PNG assets in the selected export folder.
- Create a Unity Canvas or Prefab root.
- Convert manifest coordinates to Unity UI coordinates.
- Create UI nodes according to source layer hierarchy.
- Set `RectTransform` position, size, pivot, and anchors consistently.
- Preserve render order from PSD.
- Generate ordinary image nodes for raw PNG resources.

### Hierarchy Management

The generated hierarchy should be clean and usable.

Recommended default hierarchy:

- Use `layerPath` to preserve PSD grouping where possible.
- Name nodes from manifest semantic names, not only raw file names.
- Keep major groups such as `Bg`, `Panel`, `Btn`, `Icon`, `Font`, `Effect`, if present.
- Allow empty grouping nodes for control and animation.
- Do not flatten everything into one list.

Potential output:

```text
Canvas
  UI_Login
    Bg
      Bg_Login
    Panel
      Panel_Main
        Btn_Start
        Btn_Close
        Font_Title
```

### Prefab Replacement

The plugin should support replacing manifest items with existing Unity Prefabs.

Examples:

- `Btn_Confirm` becomes `Assets/UI/Prefabs/Common/ButtonConfirm.prefab`.
- `Btn_Close` becomes an existing close button component.
- Common tabs, toggles, labels, resource counters, or item slots can be instantiated as project Prefabs.

Replacement rules can be driven by:

- `prefabHint` in manifest.
- Asset type, e.g. `Btn`.
- Name patterns.
- Layer path patterns.
- Manual mappings configured inside Unity.
- A project-level Prefab Registry.

Prefab instances should still inherit manifest-driven transform placement.

## Key Technical Challenge

File names are not stable identity.

The user explicitly expects that artists may rename PNG files after export. Therefore the plugin cannot rely only on `name` or `file` in manifest.

The robust model is:

> JSON should not only record "what the image is called"; it should record "who this asset is".

## Recommended Asset Identity System

Add stronger asset identity fields to future PSD Auto Slicer manifest output.

Recommended manifest item fields:

```json
{
  "id": "psd_9f2a_layer_1038",
  "assetType": "Btn",
  "layerPath": "Panel/Btn/Confirm",
  "rect": { "x": 310, "y": 612, "w": 188, "h": 72 },
  "zIndex": 42,
  "fileName": "Btn_Confirm.png",
  "sha256": "exact-content-hash",
  "visualHash": "perceptual-hash",
  "prefabHint": "ButtonConfirm"
}
```

Recommended fields:

- `id`
  Stable asset/layer identity. Should not change when the file is renamed.
- `layerId`
  Photoshop layer ID where available.
- `assetType`
  Semantic type such as `Bg`, `Panel`, `Btn`, `Icon`, `Line`, `Font`, `Banner`, `Asset`.
- `layerPath`
  Original PSD hierarchy path.
- `rect`
  Original bounds in PSD coordinate space.
- `zIndex`
  Explicit render order.
- `fileName`
  Original exported file name.
- `sha256`
  Exact content hash for unchanged PNGs after rename.
- `visualHash`
  Perceptual hash for similar images after re-export or compression.
- `prefabHint`
  Optional hint for prefab replacement.
- `textContent`
  Useful if later converting text layers to TextMeshPro.
- `sourceCanvasWidth`, `sourceCanvasHeight`
  Better placed at manifest root, but item-level normalized values may also help.

## Matching Strategy After Rename/Delete

Unity import should scan current PNG files and build an index.

Recommended matching priority:

1. Manual map
   Highest priority. User-confirmed mappings should override all automatic decisions.
2. Prefab Registry
   If the item should be replaced by an existing Prefab, instantiate Prefab and do not require PNG.
3. Exact hash
   If current PNG `sha256` matches manifest item `sha256`, treat as same image even if renamed.
4. Visual hash
   If image changed slightly, compare perceptual hash plus dimensions.
5. Structural signature
   Use `layerPath`, `rect`, `assetType`, and `zIndex` to resolve duplicated or ambiguous candidates.
6. Fuzzy name fallback
   Use only as a weak fallback, never as the primary identity.
7. Manual review
   If confidence is low, show a review panel.

Expected outcomes:

- `sprite`
  Use current PNG as a Sprite/Image node.
- `prefab`
  Instantiate existing Prefab.
- `skip`
  Do not generate a visible node.
- `placeholder`
  Generate an empty control node or missing-resource placeholder.
- `manual`
  Await user confirmation.

## Unity Editor MVP Scope

MVP should prove the pipeline without solving every production edge case.

MVP features:

- Editor menu item, e.g. `Tools/PSD Auto Slicer/Restore UI`.
- Select a folder containing `export_manifest.json`.
- Drag a folder from Unity Project into the `Export Folder` field.
- Parse manifest.
- Import or locate PNG files by current manifest file names.
- Match renamed PNGs by `sha256` when available.
- Fall back to same-folder unique-size matching.
- Create a Canvas or Prefab.
- Convert PSD coordinates to Unity UI coordinates.
- Create Image nodes.
- Build hierarchy from `layerPath`.
- Create the 10 root control nodes: `Bg`, `Top`, `TopLeft`, `TopRight`, `Left`, `Center`, `Right`, `Bottom`, `BottomLeft`, `BottomRight`.
- Preserve item ordering.
- Save generated Prefab.
- Show a basic report: generated count, missing count, skipped count.

Current MVP limitations:

- Export folders must be inside the Unity project's `Assets` directory.
- Rename matching is robust for unchanged PNGs when `sha256` is present.
- Same-folder unique-size fallback is intentionally conservative and will not resolve ambiguous duplicate sizes.
- Does not automatically replace nodes with project Prefabs yet.
- Does not convert text layers to TextMeshPro yet.
- Does not update an existing generated Prefab in place yet; it currently saves a new generated Prefab path.

## Production Version Scope

Production version should add:

- Asset identity matching via `assetId`, `sha256`, and `visualHash`.
- PNG scanning and matching confidence scores.
- Manual mapping UI for missing or ambiguous items.
- Prefab Registry.
- Rule-based component replacement.
- Missing asset placeholders.
- Skip rules.
- Rebuild/update existing Prefab without destroying user-added scripts where possible.
- Export/import mapping file, likely `resource_map.json`.
- Optional TextMeshPro conversion for text layers if upstream manifest includes text content.

## Suggested `resource_map.json`

This file can live beside `export_manifest.json` or inside Unity project settings.

Example:

```json
{
  "psd_9f2a_layer_1038": {
    "mode": "sprite",
    "asset": "Assets/UI/Login/Btn_Start_Normal.png"
  },
  "psd_9f2a_layer_1039": {
    "mode": "prefab",
    "prefab": "Assets/UI/Prefabs/Common/ButtonConfirm.prefab"
  },
  "psd_9f2a_layer_1040": {
    "mode": "skip"
  },
  "psd_9f2a_layer_1041": {
    "mode": "placeholder",
    "reason": "Deleted duplicate decorative layer"
  }
}
```

## Coordinate Conversion Notes

PSD coordinate system:

- Top-left origin.
- `x`, `y` likely refer to top-left bounds.
- Width and height from exported item.

Unity UI coordinate system:

- RectTransform usually uses center-based anchored position.
- Canvas setup should define a consistent reference resolution.

Recommended conversion:

- Manifest root should provide canvas width and height.
- Convert PSD top-left item rect to Unity centered coordinates:

```text
unityX = x + width / 2 - canvasWidth / 2
unityY = canvasHeight / 2 - y - height / 2
```

Recommended pivot:

- Default pivot: `(0.5, 0.5)`.
- Anchor: center by default for faithful reconstruction.
- For adaptive layouts, a later pass can convert selected groups to anchors.

## Recommended Development Phases

### Phase 1: MVP Restoration

- Build Unity Editor importer.
- Read current manifest.
- Generate Canvas/Prefab from file-name matched PNGs.
- Preserve hierarchy and approximate visual layout.

Success criteria:

- A sliced PSD can be reconstructed into Unity with visually recognizable layout.
- Missing assets are reported clearly.

### Phase 2: Manifest Enhancement

- Update PSD Auto Slicer manifest output with:
  - `assetId`
  - `layerId`
  - `assetType`
  - `zIndex`
  - root `sourceCanvasWidth/sourceCanvasHeight`
  - `sha256`
  - optional `visualHash`
  - optional `prefabHint`

Success criteria:

- Renamed PNGs can still match via hash.
- Repeated resources can still be assigned to correct nodes via layer identity and rect.

### Phase 3: Mapping Review UI

- Add Unity Editor window that shows:
  - Matched items.
  - Missing items.
  - Duplicate candidates.
  - Low-confidence matches.
  - Prefab replacement candidates.

Success criteria:

- Artists or technical artists can fix mappings without editing JSON manually.

### Phase 4: Prefab Registry and Production Rebuild

- Add rules for replacing generated Image nodes with existing UI Prefabs.
- Support update/rebuild workflow.
- Keep generated hierarchy stable for programmers and motion designers.

Success criteria:

- Common UI components can be automatically instantiated.
- Generated UI is maintainable, not just visually close.

## Risks and Open Questions

Main risks:

- Old manifests do not include stable IDs or image hashes.
- Deleted assets may be intentional or accidental; the tool needs review states.
- Duplicate PNGs can produce identical hashes, so hash alone is not enough.
- Same-folder unique-size fallback can only work when exactly one candidate has the expected size in the expected folder.
- Prefab replacement may need project-specific conventions.
- Text layers may eventually need conversion to TextMeshPro rather than PNG.
- Nine-slice handling needs Unity Sprite import setup or prefab-specific replacement.
- Updating an already edited Prefab is harder than first-time generation.

Open questions:

- Should the generated output be a Scene, a Prefab, or both?
- Should hierarchy preserve PSD exactly or reorganize by semantic categories?
- What Unity version and UI stack will be targeted: UGUI, UI Toolkit, or both?
- Should text remain PNG initially, or should text layers become TextMeshPro?
- Where should project-level Prefab Registry live?
- Who owns mapping review: artist, TA, programmer, or motion designer?

## Relationship to PSD Auto Slicer

This product should remain independent.

PSD Auto Slicer responsibilities:

- Slice PSD into PNGs.
- Generate `export_manifest.json`.
- Optionally add stronger metadata for identity and Unity reconstruction.

Unity UI Restore Plugin responsibilities:

- Read PNGs and manifest.
- Resolve renamed/deleted/replaced resources.
- Generate Unity UI hierarchy.
- Replace suitable nodes with existing Prefabs.
- Provide review and mapping tools.

The two tools should communicate through manifest and mapping files, not through direct runtime coupling.

## One-Line Principle

Do not let JSON only remember what the image is called.

Let JSON remember who the asset is, where it belongs, and whether it can be replaced by a Prefab.
